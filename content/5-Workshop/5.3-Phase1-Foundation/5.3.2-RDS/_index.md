## Objective of this Lab

In this lab, you will deploy the database infrastructure for the **Mini Social Network** application using **AWS CloudFormation**.

To maximize security and optimize operational costs, the database will be deployed entirely within **Private Subnets**. Additionally, an **EventBridge Scheduler** will automatically start and stop the database during predefined working hours, helping reduce AWS costs in development and educational environments.

---

## Learning Objectives

After completing this lab, you will be able to:

- Deploy an enterprise-grade **Amazon RDS for SQL Server**
- Secure the database inside **Private Subnets**, allowing access only from the backend application running on Amazon ECS
- Configure **Amazon EventBridge Scheduler** to automatically start and stop the database
- Enable automatic database backups using **Backup Retention**
- Protect data using **Storage Encryption**

---

## AWS Services Used

| Service | Purpose |
|---------|---------|
| Amazon RDS (SQL Server) | Managed relational database for the application |
| DB Subnet Group | Places the database inside private subnets |
| Security Group | Controls inbound access to the database (port 1433) |
| Amazon EventBridge Scheduler | Automatically starts/stops the database on a schedule |
| IAM Role | Grants EventBridge permission to manage the RDS instance |
| AWS CloudFormation | Infrastructure as Code for automated provisioning |

# Step 2 – Deploy the Database Stack

Deploy the database infrastructure using the AWS CloudFormation Console.

Navigate to:

**CloudFormation → Stacks → Create stack → With new resources (standard)**

---

## Upload the Template

Under **Prerequisite**, choose:

1. Select **Upload a template file**
2. Select **final-minisocial-db.yaml**

![Upload Database Template](/images/5-Workshop/5.3-Phase1-Foundation/Create_Stack1.png)
<center><i>Uploading the database CloudFormation template.</i></center>

Click **Next**.

---

## Configure the Stack

Enter the following stack name:

```text
MiniSocial-Database-Stack
```

Fill in the parameters collected from **Stack 1**:

| Parameter | Source |
|-----------|--------|
| VpcId | From Stack 1 Outputs |
| PrivateSubnetComputeId | From Stack 1 Outputs |
| PrivateSubnetDataId | From Stack 1 Outputs |
| BackendSecurityGroupId | From Stack 1 Outputs |
| DBMasterPassword | Enter a strong password |

{{% notice warning %}}
Do **not** use the characters `/` or `@` in **DBMasterPassword**, as they may cause connection issues when configuring the JDBC URL later.
{{% /notice %}}

![Configure Stack Parameters](/images/5-Workshop/5.3-Phase1-Foundation/Create_Stack1_2.png)
<center><i>Entering stack parameters from Stack 1 Outputs.</i></center>

Continue clicking **Next** through the remaining configuration pages.

Review all settings carefully and click **Submit**.

{{% notice info %}}
Deployment usually takes **5–10 minutes**. Do not close the browser tab while the stack is being created.
{{% /notice %}}

## Verify EventBridge Scheduler

Navigate to **Amazon EventBridge Scheduler**.

Confirm that the following schedules have been created:

| Schedule | Expected Status |
|----------|-----------------|
| Auto Start Schedule | Enabled |
| Auto Stop Schedule | Enabled |

![Amazon EventBridge Scheduler](/images/5-Workshop/5.3-Phase1-Foundation/CheckEvtBridge.png)
<center><i>Verify that both <strong>Auto Start</strong> and <strong>Auto Stop</strong> schedules are in the <strong>Enabled</strong> state.</i></center>

---

## Verify Time Zone Configuration

Open either schedule and verify that the configured time zone is:

```text
Asia/Ho_Chi_Minh
```

This ensures that the database starts and stops according to Vietnam local time.

---

{{% notice info %}}
**DeletionPolicy: Snapshot**
This database stack is configured with `DeletionPolicy: Snapshot`. If the CloudFormation stack is deleted, AWS will automatically create a final database snapshot before removing the RDS instance, preventing accidental data loss.
{{% /notice %}}

## Verification Checklist

Before continuing to the next lab, verify the following:

- ✅ CloudFormation Stack status is **CREATE_COMPLETE**
- ✅ The RDS instance is deployed inside Private Subnets
- ✅ Public accessibility is disabled
- ✅ Storage Encryption is enabled
- ✅ Only **BackendSecurityGroup** is allowed to access SQL Server on port **1433**
- ✅ EventBridge Scheduler is enabled
- ✅ Database schedules are configured to stop at **22:00** and start at **07:00** (Vietnam time)

---

### Troubleshooting

{{% notice warning %}}
If the CloudFormation stack enters the **ROLLBACK_COMPLETE** state, verify the following:

- The **DBMasterPassword** does not contain unsupported characters such as `/` or `@`.
- The VPC ID, Subnet IDs, and Backend Security Group ID were copied correctly from **Stack 1**.
- The CloudFormation template was uploaded successfully.
- You are deploying in the **ap-southeast-1 (Singapore)** region.
{{% /notice %}}
